## 简介
本技术文档面向 Scm.Net 的实时通信系统，聚焦于基于 SignalR 的实时通信架构，系统化阐述 Hub 连接管理、用户会话跟踪、消息路由机制与推送流程。文档覆盖 ScmHub 的实现原理、用户状态管理、消息确认与回执机制、连接池与缓存策略，并提供 .NET、JavaScript 与移动端的客户端集成示例路径与最佳实践。同时给出性能优化建议、连接限制与故障排除指南。

## 核心组件

## 架构总览
ScmHub 作为实时通信的核心 Hub，通过 OnConnectedAsync/OnDisconnectedAsync 维护在线用户列表；业务侧通过 NasMessageService 获取 Hub 上下文，按用户或全体进行消息推送。消息以统一响应对象封装，客户端订阅对应事件名接收实时通知。

```mermaid

Hub->>Cache : 读取在线用户列表
Hub->>Hub : 解析 JWT 获取用户标识
Hub->>Cache : 更新/添加当前用户会话
Note over Hub,Cache : 在线用户列表按用户ID去重并更新
Client->>Bus : 请求发送消息(目标用户/全体)
Bus->>Cache : 查询目标用户连接ID
alt 找到目标用户
Bus->>Hub : 指定连接发送事件
Hub-->>Client : ReceiveXxx 事件
else 未找到
Bus-->>Client : 无响应(忽略)
end
Client-->>Hub : 断开连接
Hub->>Cache : 移除该连接对应的会话
```

## 组件详解

### ScmHub：连接管理与消息路由
- 连接建立：从查询参数提取 access_token，解析 JWT 获取用户标识，构造 ClientUser 并写入缓存；若同一用户已有连接则替换旧连接。
- 断开清理：根据连接ID查找并移除对应会话。
- 消息路由：
  - 指定用户：通过缓存中的连接ID向特定客户端发送事件。
  - 全体广播：直接向所有客户端广播。
  - 特殊指令：如“踢出”指令，移除目标用户会话并向所有客户端广播“被踢出”事件。

```mermaid

Parse --> Build["构建 ClientUser 对象"]
Build --> LoadList["读取在线用户列表"]
LoadList --> Exists{"是否已存在同用户连接?"}
Exists --> |是| Replace["移除旧连接项"]
Exists --> |否| Keep["保持不变"]
Replace --> Append["追加新连接项"]
Keep --> Append
Append --> Save["写入缓存"]
Save --> End(["完成"])
DiscStart(["断开连接"]) --> Find["按连接ID查找用户"]
Find --> Found{"找到用户?"}
Found --> |是| Remove["从列表移除"]
Found --> |否| Skip["跳过"]
Remove --> Persist["写入缓存"]
Skip --> DiscEnd(["完成"])
Persist --> DiscEnd
```

## 性能与扩展性
- 连接限制
  - Kestrel Limits：MaxConcurrentConnections 控制最大并发连接数；MaxRequestBodySize 控制请求体大小，避免资源滥用。
  - 建议：结合业务峰值评估，适当上调并发连接上限，并开启连接空闲超时与心跳检测。
- 缓存策略
  - 在线用户列表存储在 ICacheService 中，建议使用 Redis 等分布式缓存以支持横向扩展。
  - 缓存键：统一使用 KeyUtils.ONLINEUSERS，避免硬编码。
- 消息路由
  - 按用户路由时先查缓存再定位连接，避免全量广播带来的网络压力。
  - 对高频事件建议合并或限流，防止风暴效应。
- 序列化
  - 使用 SignalRUtil 的统一 JSON 序列化选项，减少循环引用与冗余字段，提升传输效率。
- 客户端重连
  - 建议客户端实现指数退避重连与断线缓冲，保证在网络抖动场景下的稳定性。

## 故障排除指南
- 无法连接 Hub
  - 检查 CORS 配置与访问令牌传递是否正确。
  - 确认 appsettings.json 中 Kestrel 端口与防火墙放行。
- 连接后无消息
  - 核查在线用户列表是否写入缓存，确认用户ID与连接ID匹配。
  - 检查事件名是否与客户端订阅一致（如 ReceiveNasMessage、ReceiveMessage、ReceiveNotice 等）。
- 踢出无效
  - 确认 Hub 的踢出方法被调用且广播事件名正确。
- 性能问题
  - 检查 MaxConcurrentConnections 是否过低，必要时上调。
  - 使用分布式缓存替代内存缓存，避免多实例间的会话不一致。
- 客户端集成
  - .NET 客户端示例参考：[ClientExample.md:21-78](file://Nas.Server/Msg/ClientExample.md#L21-L78)
  - JavaScript 客户端示例参考：[ClientExample.md:16-19](file://Nas.Server/Msg/ClientExample.md#L16-L19)

## 结论
Scm.Net 的实时通信系统以 ScmHub 为核心，结合缓存与业务服务实现了可靠的连接管理、用户会话跟踪与消息路由。通过统一响应包装与序列化工具，系统在易用性与性能之间取得平衡。生产环境中建议采用分布式缓存、合理设置连接限制与客户端重连策略，以获得更稳健的实时体验。
